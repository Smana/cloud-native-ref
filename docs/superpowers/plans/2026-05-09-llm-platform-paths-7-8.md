# LLM Platform Paths 7+8 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship LoRA adapter serving on `xplane-qwen-coder` (composition `v0.6.0`, two HF-published adapters, AI Gateway routing, eval coverage, README docs) and append a per-tenant FinOps observability paragraph to `docs/llm-platform-future-paths.md`.

**Architecture:** Extend the `inference-service` composition with a new optional `loraAdapters: [LoraAdapter]` field. Empty/absent list = current behavior (regression-safe). Non-empty list emits `--enable-lora --max-loras --max-lora-rank --lora-modules` flags into vLLM args, extends the preload Job to download each adapter into the existing S3 Files PVC at `/models/loras/<name>/`, and adds per-adapter AIGatewayRoute matchRules. Path 8 is documentation only.

**Tech Stack:** KCL (composition), Crossplane v2 (`m.upbound.io` namespaced MRs), Flux, vLLM 0.8.5, Envoy AI Gateway, KEDA, S3 Files (EFS-CSI).

**Branch:** `wip/self-hosted-llm-platform-draft` (PR #1434). All changes commit onto this branch — no separate PR.

**Spec:** [`docs/superpowers/specs/2026-05-09-llm-platform-paths-7-8-design.md`](../specs/2026-05-09-llm-platform-paths-7-8-design.md).

### Task 8: Extend preload command to download adapters

**Files:**
- Modify: `infrastructure/base/crossplane/configuration/kcl/inference-service/main.k`

The preload command in main.k:594-628 currently has two `exit 0` short-circuits for the warm-base path that prevent adapter downloads from running when the base is already present. We add adapter downloads as an unconditional final phase, replacing those two `exit 0` lines with a `BASE_DONE=1` flag and gating the actual base download on it.

- [ ] **Step 1: Update `_preloadCommand` body**

Replace the entire `_preloadCommand` string (main.k lines 594-628) with this version:

```python
    _loraDownloadCommands = ["hf download " + a.repository + " --revision " + (a.revision or "main") + " --local-dir /models/loras/" + a.name + " --max-workers 1" for a in _loraAdapters]
    _loraDownloadSnippet = ("\necho '[preload] Downloading LoRA adapters...'\nmkdir -p /models/loras\n" + "\n".join(_loraDownloadCommands) + "\n") if _loraAdapters else ""

    _preloadCommand = """set -euo pipefail
DIR="/models/""" + _modelRevision + """"
MARKER="$DIR/.preload-complete"
WANT=\"""" + _modelRepository + """@""" + _modelRevision + """\"
BASE_DONE=0
# `find` returns non-zero when the directory does not exist (fresh
# preload, never run before). Combined with `set -e -o pipefail` this
# would abort the script before we even reach `hf download`. Guard the
# call so a missing cache dir simply yields INCOMPLETE_COUNT=0.
if [ -d "$DIR/.cache/huggingface/download" ]; then
  INCOMPLETE_COUNT=$(find "$DIR/.cache/huggingface/download/" -maxdepth 1 -name '*.incomplete' | wc -l)
else
  INCOMPLETE_COUNT=0
fi
# `.incomplete` files trump every other signal: even a matching marker
# is wrong if the xet cache says a shard never finished. Earlier code
# baked a bad marker after a stalled download and the wrong marker
# would otherwise short-circuit forever — clean those up explicitly
# before the marker check.
if [ "$INCOMPLETE_COUNT" -gt 0 ]; then
  echo "[preload] $INCOMPLETE_COUNT .incomplete file(s) in xet cache — clearing stale marker, will resume download."
  rm -f "$MARKER"
elif [ -f "$MARKER" ] && [ "$(cat "$MARKER" 2>/dev/null)" = "$WANT" ]; then
  echo "[preload] $WANT already complete (marker) — skipping base download."
  BASE_DONE=1
elif [ -f "$DIR/config.json" ]; then
  echo "[preload] $WANT files already in bucket (config.json present, no .incomplete cache) — recording marker, skipping base download."
  printf '%s' "$WANT" > "$MARKER.tmp" && mv "$MARKER.tmp" "$MARKER" || true
  BASE_DONE=1
fi
export PATH="/tmp/pip/bin:$PATH"
export PYTHONPATH="/tmp/pip"
pip install --quiet --target /tmp/pip --upgrade --no-cache-dir huggingface_hub
if [ "$BASE_DONE" -ne 1 ]; then
  hf download """ + _modelRepository + """ --revision """ + _modelRevision + """ --local-dir "$DIR" --max-workers 1
  printf '%s' "$WANT" > "$MARKER.tmp" && mv "$MARKER.tmp" "$MARKER"
fi""" + _loraDownloadSnippet + """echo '[preload] Done.'
"""
```

**Diff summary:**
- Added `_loraDownloadCommands` list comprehension and `_loraDownloadSnippet` string above `_preloadCommand` (place these in the same `if _preloadEnabled:` block, just before the `_preloadCommand` assignment).
- Inserted `BASE_DONE=0` near top.
- Replaced the two `exit 0` lines with `BASE_DONE=1`.
- Wrapped the `hf download` of base inside `if [ "$BASE_DONE" -ne 1 ]; then ... fi` so the script always continues to the LoRA snippet.
- Moved `pip install` outside the gate (always needed, since adapter downloads need it too even when base is warm).
- Appended `_loraDownloadSnippet` (which is empty string when `loraAdapters` is empty — keeping current behavior identical for non-LoRA claims).

```markdown
## 7. Per-tenant FinOps observability

Self-hosted LLM serving collapses the "infra cost vs. token cost" boundary
onto a single bill, which makes per-consumer accounting structurally
important. The 2026 framing (cardinality climbs faster than on a regular
service because every FinOps dimension wants to be a label; volume scales
with token throughput, not request rate; alerting moves from "the service
is down" to "tenant X exceeded their hourly budget") describes the shape
of the problem.

**Approach** (compatible with the foundation-showcase non-goal of *no
production multi-tenant serving*): tenant identity comes from a static
`x-tenant` request header (default `anonymous`). Envoy AI Gateway extracts
it via stats config and emits a `tenant` label on its existing token
counters. A new VMRule defines per-tenant burn-rate alerts on
`sum by(tenant) (rate(envoy_ai_gateway_tokens_total[5m]))`. A new Grafana
dashboard panel set surfaces tokens-per-tenant, cost-per-tenant
(constant-times-tokens approximation), and top-talker over time.

**Cost**: ~zero infra cost; the only meaningful overhead is metric
cardinality. Cap at ~10 tenants for the showcase via VMAgent
`metric_relabel_configs` allowlist.

**Trigger**: when any real (or simulated) workload routes through the
platform with multiple addressable consumers — including LoRA adapter
names (already shipped via the `loraAdapters` field on `inference-service`),
used as proxy "tenants" to demo the cost-attribution story without standing
up auth.

**What this is *not***: tenant authentication, quotas, fairness scheduling,
or rate limiting. Those remain non-goals for this platform's posture; doing
them would warrant amending the foundation-showcase spec, not this
future-path entry.
```
