# Spec-Driven Development (SDD)

Specifications for non-trivial changes to the cloud-native-ref platform.

## Overview

SDD ensures thoughtful planning before implementation, reducing rework and catching design issues early. Inspired by [GitHub Spec Kit](https://github.com/github/spec-kit) and refined for platform engineering, the workflow separates **WHAT** (the contract), **HOW** (the design), **WHEN/WHO** (execution), and the **DECISION TRAIL** (clarifications) into distinct, durable artifacts.

```
/spec  →  /spec-research (optional)  →  /clarify  →  /validate  →  /analyze  →
Implement  →  /create-pr  →  Auto-archive (with SUMMARY)  →  /verify-spec  →  /debug-spec (if needed)
```

**Key documents**:
- [Platform Constitution](./constitution.md) — non-negotiable principles all specs must follow (also surfaced by the `platform-constitution` reference-skill)
- [Architecture Decision Records](../decisions/) — cross-cutting technology choices

## When to create a spec

Run `/spec` when making:

| Change Type | Examples |
|-------------|----------|
| **composition** | New KCL module, new XRD, Crossplane patterns |
| **infrastructure** | New OpenTofu stack, VPC changes, EKS upgrades |
| **security** | Network policies, RBAC, PKI, secrets management |
| **platform** | Multi-component features, observability, GitOps changes |

## When to skip specs

- Version bumps (Renovate PRs)
- Documentation-only changes
- Single-file bug fixes
- Minor configuration tweaks
- HelmRelease value changes

## Spec directory layout (4 artifacts)

Each spec lives in its own directory under `docs/specs/NNN-slug/`:

```
docs/specs/NNN-slug/
├── spec.md            ← WHAT — requirements, user stories, FR-XXX, SC-XXX. Frozen after approval.
├── plan.md            ← HOW — design, resources, 4-persona review checklist. May evolve.
├── tasks.md           ← WHEN/WHO — T001+ checklist. Updated commit-by-commit.
├── clarifications.md  ← Append-only decision log (CL-1, CL-2, …). Never overwritten.
└── (optional)
    ├── research.md       ← Written by /spec-research before filling spec
    ├── debug/<slug>.md   ← Persistent debug session (from /debug-spec)
    └── (post-merge:)
        ├── SUMMARY.md         ← Auto-generated by spec-archive.yaml
        └── VERIFICATION.md    ← Written by /verify-spec
```

**Rationale for the split**: each artifact has a different edit cadence. Mixing them (the old single-`spec.md` approach) meant requirements got mutated when implementation evolved, losing the original contract. With 4 files, "what was promised" stays separate from "what we built."

After merge the directory moves to `docs/specs/done/YYYY-Qn/NNN-slug/` (quarterly bucketing keeps `done/` browsable as the repo ages).

## Workflow

```
┌──────────┐  ┌──────────────┐  ┌────────────┐  ┌─────────┐  ┌──────────┐
│  /spec   │→ │ /spec-research│→│  Fill 4    │→ │ /clarify │→ │ /validate│
│          │  │  (optional)   │ │  artifacts │  │ → CL-N    │  │  & /analyze│
│ creates  │  │ → research.md │  │            │  │ entries   │  │            │
│ 4 files  │  │               │  │            │  │           │  │            │
└──────────┘  └──────────────┘  └────────────┘  └─────────┘  └──────────┘
                                                                       │
                                                                       ▼
┌────────────┐  ┌──────────────┐  ┌──────────────────┐  ┌──────────────┐
│ /create-pr │→ │ Merge to main│→ │ Archive workflow │→ │ /verify-spec │
│            │  │              │  │ - generates      │  │ → VERIFICATION
│ references │  │              │  │   SUMMARY.md     │  │              │
│ spec dir   │  │              │  │ - moves to       │  │              │
│            │  │              │  │   done/YYYY-Qn/  │  │              │
└────────────┘  └──────────────┘  └──────────────────┘  └──────────────┘
```

### Step 1: Create the spec

```bash
/spec composition "Add Valkey caching"
/spec infrastructure "Add GPU node pool"
```

Calls `scripts/sdd/create-spec.sh` which:
- Creates the GitHub issue with `spec:draft` label
- Creates the directory `docs/specs/NNN-slug/`
- Instantiates `spec.md`, `plan.md`, `tasks.md`, `clarifications.md` from `docs/specs/templates/`

### Step 2 (optional): Pre-deliberation research

```bash
/spec-research 003-valkey "Valkey vs Redis caching patterns + community defaults"
```

Runs in a forked Explore subagent so Context7 + WebSearch + repo scan don't burn the main context. Writes `research.md` next to the spec — it surfaces existing patterns, ecosystem options, and "don't hand-roll" guidance before you even start filling `spec.md`.

### Step 3: Fill the artifacts

| File | What you write |
|------|----------------|
| `spec.md` | Summary, problem, user stories with Gherkin scenarios, FR-XXX, SC-XXX (falsifiable), non-goals. Mark unknowns with `[NEEDS CLARIFICATION: ...]`. |
| `plan.md` | Design (XRD, generated resources, dependencies, alternatives considered), implementation notes, 4-persona Review Checklist. |
| `tasks.md` | Phased T001+ checklist (prerequisites, implementation, validation). |
| `clarifications.md` | Leave empty until `/clarify` runs. |

### Step 4: Resolve clarifications

```bash
/clarify
```

For each `[NEEDS CLARIFICATION: ...]` marker, presents 2–3 structured options using the [4-perspective decision framework](../../.claude/skills/clarify/references/decision-framework.md). On answer:
- Appends a new `## CL-N` entry to `clarifications.md` with options, decision, rationale, references.
- Replaces the marker in `spec.md` (or `plan.md`) with `CL-N — <one-line summary>` so the artifact references the durable record.

The deliberation is permanent — six months later, "why did we pick option A?" always has an answer.

### Step 5: Validate completeness

```bash
/validate
```

Runs `scripts/validate-spec.sh` against the directory. Checks:
- All 4 artifacts present
- `spec.md`: required sections, GitHub issue link, FR/SC counts, falsifiable SCs (no vague adjectives)
- `plan.md`: Design + Review Checklist sections; checklist ≥ 75%
- `tasks.md`: structured `T001` IDs
- `clarifications.md`: append-only `## CL-N` format, no duplicate IDs
- No unresolved `[NEEDS CLARIFICATION]` or inline `[CLARIFIED:]` (decisions belong in `clarifications.md`)

### Step 6: Cross-artifact analysis

```bash
/analyze
```

Detects problems single-artifact validation misses: FR-XXX with no implementing task, SC-XXX with no verification path, constitution violations in `plan.md`, terminology drift between artifacts, duplicate requirements, stale CL-N references. Verdict: **BLOCK** (CRITICAL findings) / **PASS WITH WARNINGS** / **PASS**.

### Step 7: Implement

Work through `tasks.md`, checking off `T001`, `T002`, … as commits land. The `crossplane-validator` skill auto-runs for KCL changes; `platform-constitution` auto-loads to keep non-negotiable rules in scope.

### Step 8: Create PR

```bash
/create-pr
```

Auto-detects the spec directory in the diff; references the issue (`Implements #XXX`); generates a structured PR body with mermaid diagram and file table.

### Step 9: Auto-archive on merge

`.github/workflows/spec-archive.yaml` runs after the PR merges:

1. Locates `docs/specs/NNN-slug/` from the PR body.
2. Runs `scripts/sdd/generate-summary.sh` to write a `SUMMARY.md` with PR metadata (commits, file diffstat by area, SC snapshot, deviations from plan).
3. Moves the entire directory into the quarterly bucket: `docs/specs/done/YYYY-Qn/NNN-slug/`.
4. Closes the linked issue with `spec:done` label.

### Step 10: Post-deploy verification

```bash
/verify-spec docs/specs/done/2026-Q2/NNN-slug
```

Deploys the example, queries Flux/VictoriaMetrics/VictoriaLogs MCPs, writes `VERIFICATION.md` with one row per `SC-XXX` (PASS / FAIL / MANUAL). Closes the loop on whether the merged feature actually meets its acceptance criteria.

### Step 11 (if needed): Persistent debugging

```bash
/debug-spec docs/specs/done/2026-Q2/NNN-slug sqs-reconcile-loop
```

Creates `<spec-dir>/debug/sqs-reconcile-loop.md` with frontmatter status, current focus, hypotheses tracked, evidence log, eliminated paths. Survives session resets so multi-day investigations don't lose state.

## Phased specs (large features only)

For features that genuinely require multiple PRs (e.g., the QueueInstance pattern that spanned issues #1300–1304: prereqs → composition → packaging → validation → docs), keep one spec directory but break execution into sub-phases:

```
docs/specs/005-queueinstance/
├── spec.md             ← One set of requirements for the whole feature
├── plan.md             ← Overall design + phase map
├── clarifications.md   ← Cross-phase decisions
├── tasks.md            ← Cross-phase task overview
└── phases/
    ├── 1-prereqs/
    │   ├── tasks.md          ← Phase-local task list
    │   └── SUMMARY.md        ← Created when phase PR merges
    ├── 2-composition/
    │   ├── tasks.md
    │   └── SUMMARY.md
    └── ...
```

One GitHub issue per phase (matches existing `[Phase N]` issue pattern), all linked back to the parent spec issue with `Depends on #<parent-issue>`. Apply phase-specific labels (`phase:1-prereqs`, `phase:2-composition`, …).

Use phased specs only when the work genuinely cannot ship in one PR. Most composition work fits a single spec.

## Spec template structure

Every artifact has a template under [`templates/`](templates/):

| Template | Sections | Cadence |
|----------|----------|---------|
| `spec.md` | Metadata, Summary, Problem, User Stories (Gherkin), Requirements (FR-XXX), Success Criteria (SC-XXX, falsifiable), Open questions, References | **Frozen after approval** |
| `plan.md` | Design (API, resources, alternatives), Implementation notes, **Review Checklist (4 personas)**, optional Phases table, References | Evolves with implementation |
| `tasks.md` | Phased `T001+` checklist + Deviations log | Updated per commit |
| `clarifications.md` | Append-only `## CL-N` entries (options + decision + rationale + references) | Append-only, never edited |
| `SUMMARY.md` | Auto-generated by archive workflow: commits, files, SC snapshot, deviations | Written once on merge |

## User Stories (Gherkin-style)

```markdown
### US-1: Deploy Cached Application (Priority: P1)

As a **developer**, I want to deploy an app with managed caching,
so that I can improve response times without managing Redis myself.

**Acceptance Scenarios**:
1. **Given** an App claim with `cache.enabled: true`,
   **When** the composition reconciles,
   **Then** a Valkey instance is created in the same namespace.
2. **Given** a Valkey instance is ready,
   **When** the app starts,
   **Then** the CACHE_URL environment variable is injected.
```

## Review personas (lives in plan.md)

The 4-persona Review Checklist enforces multi-angle scrutiny before implementation begins.

### Project Manager
- Problem statement is clear and specific
- User stories capture real user needs
- Acceptance scenarios are testable
- Scope is well-defined (goals AND non-goals)
- Success criteria are measurable

### Platform Engineer
- Design follows existing patterns (App, SQLInstance, EPI as references)
- API is consistent with other compositions
- Resource naming follows `xplane-*` convention
- KCL avoids mutation pattern (function-kcl #285)
- Examples provided (basic + complete)

### Security & Compliance
- Zero-trust networking (CiliumNetworkPolicy defined)
- Least-privilege RBAC
- Secrets via External Secrets (no hardcoded credentials)
- Security context enforced (non-root, read-only FS where possible)
- IAM policies scoped to `xplane-*` resources (if AWS)

### SRE
- Health checks defined (liveness, readiness probes)
- Observability configured (metrics → VictoriaMetrics, logs → VictoriaLogs)
- Resource requests + limits appropriate
- Failure modes documented
- Recovery / rollback path clear

## Clarification markers and CL-N references

In a fresh `spec.md` (or `plan.md`):

```markdown
- [ ] [NEEDS CLARIFICATION: Should cache support cross-namespace access?]
```

After running `/clarify`:

```markdown
- [x] CL-3 — Cross-namespace cache access?
```

The full deliberation lives in `clarifications.md`:

```markdown
## CL-3 — 2026-04-18 — Should cache support cross-namespace access?

**Asked by**: Spec author
**Context**: Some apps want to share cache instances across teams; security policy is namespace-scoped.

**Options considered**:
| Option | Answer | Pros | Cons |
|--------|--------|------|------|
| A | Namespace-scoped only | Aligns with zero-trust; simpler RBAC | Apps in different ns must run their own cache |
| B | Allow cross-ns via explicit allow-list | Resource sharing | Complicates network policy; new RBAC surface |

**Decision**: A — namespace-scoped only
**Rationale**: Matches zero-trust default; cross-namespace exception can be added later without breaking existing consumers.
**Decided by**: Platform team (/clarify session, 2026-04-18)
**References**: constitution.md → "zero-trust networking"
```

## GitHub issue labels

| Label | Description | Transition |
|-------|-------------|------------|
| `spec:draft` | Initial state, spec is being filled | Auto-added by `/spec` |
| `spec:implementing` | Work has started | Manual: `gh issue edit XXX --remove-label spec:draft --add-label spec:implementing` |
| `spec:done` | Spec archived after PR merge | Auto-added by archive workflow |
| `spec:regression` | Post-merge verification surfaced an issue (recommended convention) | Manual when `/verify-spec` reports failures |

```bash
gh issue list --label "spec:draft"        # Specs needing work
gh issue list --label "spec:implementing" # Specs in progress
gh issue list --label "spec:done"         # Completed specs
```

## Auto-archive on PR merge

The archive workflow (`.github/workflows/spec-archive.yaml`):

1. Detects spec directory in PR body (e.g., `docs/specs/001-feature-name`).
2. Runs `scripts/sdd/generate-summary.sh` to populate `SUMMARY.md` from PR + git metadata.
3. Moves the directory into `docs/specs/done/YYYY-Qn/` (quarterly bucket based on merge date).
4. Closes the linked issue with `spec:done` label.

For this to work, the PR body must contain the spec directory path. The `/create-pr` skill handles this automatically.

## Platform constitution

All specs must comply with the [Platform Constitution](./constitution.md). Key principles:

- Resource naming: `xplane-*` prefix
- KCL: no mutation after creation (function-kcl #285)
- Security: zero-trust, least privilege, External Secrets
- IAM: EKS Pod Identity, scoped to `xplane-*` resources
- Validation: Polaris ≥ 85, kube-linter, Datree
- Observability: VictoriaMetrics for metrics, VictoriaLogs for logs

The `platform-constitution` reference-skill auto-loads these rules whenever Claude is editing files under `infrastructure/`, `security/`, `observability/`, `tooling/`, or `docs/specs/`.
